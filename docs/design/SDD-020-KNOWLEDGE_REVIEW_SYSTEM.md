# SDD-020

# Engineering Knowledge Review System

Version: 1.0

Status: Draft

---

# Purpose

The Engineering Knowledge Review System defines how proposed engineering knowledge becomes trusted repository knowledge.

Every Engineering Object entering an OEP Repository shall pass through a structured review process.

Review is an engineering activity.

Not an AI activity.

---

# Core Principle

Artificial Intelligence proposes.

Engineers approve.

Repository truth is established by engineering review.

---

# Review Pipeline

Proposal

↓

Evidence

↓

Comparison

↓

Decision

↓

Validation

↓

Commit

---

# Proposal

Every proposal contains:

* Engineering Object
* Object Type
* Confidence
* Source Evidence
* Repository Matches
* Suggested Relationships

Nothing is accepted automatically.

---

# Evidence Panel

Every proposal shall display:

- Source Material
- Source Page
- Original Text
- Original Image
- OCR Text (when available)
- Supporting Tables
- Evidence Objects
- Evidence Regions
- Related Specifications

The engineer must always be able to answer:

"Why did OEP propose this?"

---

# Repository Comparison

Every proposal is compared against:

Existing Components

Existing Procedures

Existing Specifications

Existing Images

Existing Documents

Possible outcomes:

No Match

Possible Match

Exact Match

Conflict

---

# Decision Options

Every proposal supports:

Accept

Reject

Merge

Edit

Postpone

Duplicate

No decision is irreversible until Commit.

---

# Validation

Before Commit the system validates:

Missing references

Broken relationships

Duplicate names

Conflicting specifications

Invalid object types

Missing evidence

Validation results remain attached to the Knowledge Curation Session.

---

# Confidence

Confidence assists prioritization.

Confidence never determines repository truth.

Suggested visualization:

Green

95–100%

Yellow

80–94%

Red

Below 80%

The engineer remains responsible.

---

# Review History

Every proposal records:

Reviewer

Date

Decision

Reason

Evidence Reviewed

Manual Edits

Repository Outcome

Review history becomes permanent repository knowledge.

---

# Bulk Review

Support:

Accept All High Confidence

Reject All Low Confidence

Merge Exact Matches

Clear Decisions

Bulk actions remain reviewable.

---

# Conflict Resolution

Conflicting engineering knowledge is expected.

The review system shall encourage:

Comparison

Discussion

Evidence review

Alternative representations

Repository knowledge shall evolve rather than overwrite history.

---

# Design Goals

The review experience shall feel:

Transparent

Methodical

Trustworthy

Explainable

Repeatable

The engineer should understand every repository change before approving it.

---

# Long-Term Vision

The Engineering Knowledge Review System becomes the central quality assurance process for every Engineering Object ever added to OEP.

Every repository becomes increasingly trustworthy because every piece of engineering knowledge passes through a repeatable engineering review process.
