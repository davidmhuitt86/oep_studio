# Knowledge Session Format

Introduced in Work Package 008 (STUDIO-TASK-000015 Persistent
Sessions). Documents the on-disk format for a Knowledge Curation
Session and the in-memory models it round-trips, plus the
architectural findings this work package's implementation surfaced.
See `docs/KNOWLEDGE_STUDIO.md` for the surrounding workspace/lifecycle
documentation this format supports.

Everything here is **Studio-only** — Foundation never reads or writes
any of this. "Persistence is local to Studio. Foundation shall remain
unaware of Knowledge Sessions."

---

## Local Storage Format

Root directory:

```
%APPDATA%/oep_studio/knowledge_sessions/
```

(`KnowledgeSessionStorage.root()` — falls back to `%LOCALAPPDATA%`,
then the OS temp directory, if `%APPDATA%` isn't set; in practice on a
real Windows machine `%APPDATA%` is always present.)

One subdirectory per session, named by the session's own generated ID:

```
knowledge_sessions/
  <sessionId>/
    session.json
    sources/
      <sourceId>_<originalFileName>
      <sourceId>_<originalFileName>
      ...
```

* `session.json` — the complete `KnowledgeSessionRecord` (session +
  candidates + relationship candidates + sources + review decisions),
  written with `JsonEncoder.withIndent('  ')` — human-readable, per
  this work package's explicit requirement ("Storage format shall be
  human-readable. JSON is recommended.").
* `sources/` — every attached Source Material's **copied** file (see
  § Source Material Model below for why it's a copy, not a reference
  to the original path). File names are prefixed with the source's own
  generated ID to avoid collisions between two attached files that
  happen to share an original name.

Session IDs, source IDs, candidate IDs, and relationship candidate IDs
are all generated by `KnowledgeSessionService.generateId(prefix)`:
`"$prefix-${millisecondsSinceEpoch}-${randomHex4}"` — unique enough for
a single Studio installation's sessions, not a repository-durable ID
(no repository commit has happened, so no such ID exists yet).


Every enum serializes as its Dart `.name` (`"component"`,
`"accepted"`, `"references"`, `"created"`, ...) and deserializes via
`EnumType.values.byName(...)` — no code generation, consistent with
this project's minimal-dependency philosophy. Every `DateTime`
serializes via `toIso8601String()`. `formatVersion` exists for forward
compatibility even though only `1` has ever been written; nothing
currently reads or branches on it.

### Corruption and error handling

`KnowledgeSessionStorage.load` throws `KnowledgeValidationException`
— never a raw `IOException`, `FormatException`, or `TypeError` — for:

* A missing `session.json` ("That session could not be found.").
* Malformed JSON (`FormatException` from `jsonDecode`).
* Structurally invalid JSON — a required field missing, or an
  unrecognized enum value (`TypeError`/`FormatException` from
  `KnowledgeSessionRecord.fromJson`/its nested `fromJson` calls).

`KnowledgeSessionStorage.listAll` never lets one corrupted session
block the whole Session Browser from opening — corrupted session IDs
are collected separately (`SessionBrowserListing.corruptedSessionIds`)
and shown in the browser with an explanation, rather than silently
dropped.

## Session Persistence

Every mutation to the active session's candidates, relationship
candidates, sources, or status triggers an **autosave** — there is no
separate explicit "Save" action to forget to click.
`FoundationRuntimeNotifier`'s private `_persistActiveSession()`:

1. Bumps `KnowledgeSession.lastModified` to `DateTime.now()`.
2. Builds a `KnowledgeSessionRecord` from the Connection Manager's
   current in-memory state (`knowledgeSession`, `candidates`,
   `relationshipCandidates`, `sourceMaterials`, `reviewDecisions`).
3. Calls `KnowledgeSessionStorage.save`, which writes `session.json`
   (creating the session's directory if needed).
4. On success, clears any prior `knowledgeStorageError`.
5. On a `KnowledgeValidationException`, sets `knowledgeStorageError` to
   the failure's message rather than throwing — most mutating actions
   (e.g. clicking Accept on a candidate row) have no dialog open to
   catch an exception in.

This is called as the last step of every candidate/relationship-
candidate/source-material mutation method, fire-and-forget
(`unawaited`) — the in-memory state update itself is synchronous and
immediate; the disk write happens in the background. A storage failure
never blocks the UI action that triggered it (the candidate is still
added/edited/accepted in memory even if the save fails); it only
surfaces via the Session Header's dismissible error banner.

Session-lifecycle actions that go through the Session Browser
(Open/Duplicate/Archive/Delete) are `Future<void>`-returning and
*rethrow* on failure — these are explicit, single-purpose user actions
with a dialog context available to show the error in immediately,
unlike the fire-and-forget autosave path.

## Session Browser

`lib/knowledge/sessions/session_browser_dialog.dart`, opened via the
Session Header's "Sessions" button. Lists every persisted session
(`KnowledgeSessionStorage.listAll`, sorted by `lastModified`
descending) with:

* **Open** (`FoundationRuntimeNotifier.openKnowledgeSession`) — loads
  the session's full record from disk and replaces the active session
  in memory. Same replacement semantics as creating a new session
  (the previously active session isn't lost — it's still on disk,
  since it was autosaved before being replaced).
* **Duplicate** (`duplicateKnowledgeSession`) — `KnowledgeSessionService.buildDuplicate`
  builds a fresh record (new ID, name `"Copy of <original name>"`, new
  `createdTime`/`lastModified`, status reset to `created`, candidates/
  relationship candidates/review decisions carried over unchanged,
  source `SourceMaterial.localPath` entries remapped to the new
  session's own `sources/` directory); `KnowledgeSessionStorage.duplicateSourceFiles`
  then physically copies each source file into that new directory
  before the record is saved. The two sessions are fully independent
  afterward — editing or deleting one never affects the other's files.
* **Archive / Unarchive** (`setKnowledgeSessionArchived`) — toggles
  `KnowledgeSession.archived` and re-saves; updates the in-memory
  active session too if it happens to be the one archived.
* **Delete** (`deleteKnowledgeSession`, gated behind an inline
  confirmation dialog) — `KnowledgeSessionStorage.delete` removes the
  session's entire directory (`session.json` and every attached source
  file) recursively. If the deleted session was the active one, it's
  also unloaded from memory.

Corrupted sessions are listed with an error icon and explanation
rather than omitted or blocking the list.

## Source Material Management

`lib/knowledge/models/source_material.dart` /
`source_material_type.dart`; `lib/knowledge/services/source_material_service.dart`.

Supported types, classified **by file extension only** —
`SourceMaterialType.fromExtension` — never by reading file contents:

| Type | Extensions |
|---|---|
| `pdf` | `.pdf` |
| `image` | `.png`, `.jpg`, `.jpeg`, `.gif`, `.bmp`, `.webp` |
| `markdown` | `.md`, `.markdown` |
| `text` | `.txt` |
| `other` | anything else |

"No OCR. No parsing. This work package only manages engineering
evidence" — `SourceMaterialService` copies files and records metadata;
it never reads a source's contents for classification or extraction.

`SourceMaterialService.attach(sessionId, pickedFilePath, addedBy)`:

1. Verifies the picked file exists (throws `KnowledgeValidationException`
   if not — "The selected file could not be found.").
2. Generates a source ID.
3. Copies the file into `KnowledgeSessionStorage.sourcesDirectory(sessionId)`,
   named `<sourceId>_<originalFileName>`.
4. Records `originalFileName`, the new `localPath` (the managed copy,
   not the original picked path), `type`, `sizeBytes` (of the copy),
   `importDate`, `addedBy`.

**The file is copied, not referenced, at attach time.** This is a
deliberate design choice: a session's Source Material stays available
and the session remains portable/self-contained even if the engineer
later moves, renames, or deletes the original file they picked. "Missing
source files" (this work package's error-handling requirement) refers
specifically to the *managed copy* going missing (e.g. the session
directory was tampered with outside Studio) — checked via
`SourceMaterialService.exists`, which the Source Viewer panel uses to
show an honest "Missing source file" message rather than crashing on a
missing `File`.

Removal (`SourceMaterialService.removeFile`) best-effort-deletes the
managed copy; a failure here doesn't block removing the source from
the session's list, since the in-memory/persisted record is the source
of truth for what Studio considers "attached," not the filesystem.

## Knowledge Candidate Model

```dart
class KnowledgeCandidate {
  final String id;
  final KnowledgeCandidateType type;   // component | procedure | specification | image | document
  final String name;
  final String description;
  final KnowledgeCandidateStatus status;   // pending | accepted | rejected
  final DateTime createdTime;
  final DateTime? modifiedTime;
}
```

Renamed from `EngineeringProposal` in Work Package 008 (see
`docs/KNOWLEDGE_STUDIO.md`'s Terminology note). Unchanged in shape from
Work Package 007 — still intentionally minimal (SDD-018's "Draft"
state), no AI-era fields. Name uniqueness is enforced per-session,
case-insensitively, by `KnowledgeSessionService.validateCandidateName`.

## Relationship Candidate Model

```dart
class RelationshipCandidate {
  final String id;
  final String sourceCandidateId;   // a KnowledgeCandidate.id within the same session
  final String targetCandidateId;   // a KnowledgeCandidate.id within the same session
  final RelationshipType type;      // reused from core/models/relationship_type.dart
  final String description;
  final DateTime createdTime;
  final DateTime? modifiedTime;
}
```

New in Work Package 008. Connects two `KnowledgeCandidate`s **within
the same session**, by ID — not existing repository objects; this work
package's relationship authoring is scoped to candidates that don't
exist in the repository yet. Deliberately reuses `RelationshipType`
(`lib/core/models/relationship_type.dart`) rather than inventing a
Knowledge-specific taxonomy — it already mirrors Foundation's
`oep_relationship_type_t` exactly, and a manually-authored relationship
candidate is still describing one of the same six Foundation
relationship kinds, just not yet committed. Inventing a parallel enum
here would have been exactly the kind of independent architectural
decision this work package's instructions prohibit.

**Carries no accept/reject status**, unlike `KnowledgeCandidate` — the
work package's Requirements list only specifies Create/Edit/Delete for
relationship candidates, not a review decision; every relationship
candidate that exists is included in the Commit Preview.

Validation:

* **Blocking** (`KnowledgeSessionService.validateRelationshipCandidate`,
  throws `KnowledgeValidationException`): self-reference prohibited
  (`sourceCandidateId == targetCandidateId`); both endpoints must exist
  among the session's current candidates.
* **Warned, not blocking** (`isDuplicateRelationshipCandidate`): a
  relationship with the same source/target/type already existing is
  surfaced as an inline warning in the New/Edit dialog, but submission
  is still allowed. This distinction — "prohibited" vs. "warned" — is
  drawn directly from the work package's own wording.

Deleting a candidate cascades: `FoundationRuntimeNotifier.deleteKnowledgeCandidate`
removes any relationship candidate that referenced it as source or
target, so the UI never shows a relationship pointing at a candidate
that no longer exists. `computeCommitPreview` still checks for dangling
references defensively, in case a future code path ever bypasses this
cascade.

## Commit Preview Model

```dart
class CommitPreview {
  final List<KnowledgeCandidate> newObjects;          // accepted candidates
  final List<KnowledgeCandidate> rejectedCandidates;   // rejected candidates
  final List<RelationshipCandidate> relationships;     // every relationship candidate
  final int modifiedObjectCount;   // always 0 in this work package
  final int mergedObjectCount;     // always 0 in this work package
  final List<String> validationIssues;
  final RepositoryStatistics? currentStatistics;   // unmodified; null if unavailable

  int? get projectedObjectCount;         // currentStatistics.totalObjectCount + newObjects.length, or null
  int? get projectedRelationshipCount;   // currentStatistics.relationshipCount + relationships.length, or null
}
```

New in Work Package 008 (STUDIO-TASK-000018). Computed on demand by
`KnowledgeSessionService.computeCommitPreview` from the Connection
Manager's current in-memory state — never stored, since it has no
independent existence beyond "what the current session's data would
produce right now." `validationIssues` collects: any relationship
candidate whose source/target no longer exists, and (if greater than
zero) a count of candidates still pending review. See § Architectural
Observations for why `currentStatistics` is exposed unmodified rather
than a fabricated per-category projection.

## Architectural Observations

### `KnowledgeCandidateType` does not map one-to-one onto `ObjectCategory`

This is the most significant architectural finding from Work Package
008's implementation, and per that work package's instructions
("document it and stop for architectural review rather than
introducing an independent design"), it is flagged here rather than
resolved unilaterally.

| `KnowledgeCandidateType` | Corresponding `ObjectCategory`? |
|---|---|
| `component` | `component` |
| `procedure` | `procedure` |
| `specification` | **none** |
| `image` | `image` |
| `document` | `document` |
| — | `diagram` (no Knowledge Candidate equivalent) |
| — | `project` (no Knowledge Candidate equivalent) |

`KnowledgeCandidateType` was defined in Work Package 007 as "five of
SDD-015's Layer 3 Engineering Objects — the subset Work Package 007
lists," while `ObjectCategory` mirrors Foundation's `ObjectType` enum
(Component/Document/Diagram/Procedure/Image/Project — six values, a
different five-plus-one set). This mismatch went unnoticed until this
work package's Commit Preview needed to project what committing the
session's accepted candidates would do to the repository's per-category
object counts — at which point it became a genuine blocker: Foundation
has no `Specification` object type to project a `specification`
candidate into, and there is no Knowledge Candidate type that would
ever project into `Diagram` or `Project`.

**Resolved for this work package by not fabricating an answer.**
`CommitPreview` exposes `currentStatistics` as Foundation's own,
completely unmodified `RepositoryStatistics` snapshot, plus
`projectedObjectCount`/`projectedRelationshipCount` — simple aggregate
totals (`currentTotal + acceptedCandidateCount`,
`currentRelationshipTotal + relationshipCandidateCount`) that need no
per-category mapping at all. No per-category projected breakdown is
shown, because computing one would require guessing an answer to a
question this work package's instructions explicitly say to escalate
instead of guess.

**This needs an architectural decision before a real Repository Commit
implementation can exist**, since Commit will need to know exactly
which `ObjectCategory` each accepted candidate becomes. Two directions
this could be resolved (not a recommendation — genuinely open):

1. Extend `ObjectCategory` (and Foundation's `ObjectType`) with
   `Specification`, and add `Diagram`/`Project` to
   `KnowledgeCandidateType`, so the two taxonomies become identical.
2. Keep the taxonomies distinct and add an explicit mapping table with
   a defined answer for `Specification` (e.g. commits as a
   `Document`?) and for `Diagram`/`Project` (candidates can never
   become these — is that acceptable, or does Knowledge Studio need
   Diagram/Project candidate types?).

### "Archived" as an independent field, not a new status

`KnowledgeSession.archived` is a standalone `bool`, orthogonal to
`SessionStatus`. A session's curation-workflow stage and whether it has
been archived out of the Session Browser's default view are different
lifecycle dimensions — this mirrors SDD-018's treatment of Archive for
Engineering Objects ("Archive does not imply deletion. Archived
knowledge remains searchable."), applied here to sessions. This was a
low-ambiguity decision (SDD-018 already establishes the precedent
directly) and didn't need to stop for review.

### Autosave, not an explicit Save action

Every mutation persists immediately rather than requiring an explicit
"Save" button. Reasoning: SDD-018's provenance guarantees ("nothing of
engineering significance is ever silently lost") are stronger under
autosave than under a forgettable manual save action, and Work Package
008's own requirement is unconditional ("Sessions shall survive
application restart") rather than conditional on the engineer
remembering to save. The tradeoff — background I/O on every mutation,
today only visible on failure via `knowledgeStorageError` — was judged
acceptable given how small each session's data is expected to be
(JSON metadata plus a handful of copied evidence files, not large
binary payloads Studio itself generates).

### Session Browser as a dialog, and Relationship View as a tab

Both STUDIO-TASK-000015 (Session Browser) and STUDIO-TASK-000017
(Relationship View) describe UI that this work package placed inside
existing surfaces rather than as new workspace panels:

* The **Session Browser** opens as a dialog (`showDialog`), not a
  panel — browsing/switching sessions is an occasional action, not
  something that needs permanent screen space competing with SDD-016's
  seven fixed panels.
* The **Relationship View** is a tab inside the existing Engineering
  Review panel, not an eighth panel — SDD-016 fixes the layout at
  seven panels, and the work package's Requirements list never actually
  requires a *separate* panel, only that the described actions exist
  somewhere. Knowledge Candidates and Relationship Candidates are both
  "things awaiting engineering review," so sharing one panel via a tab
  keeps them conceptually together without touching the frozen layout.

Neither required stopping for review — both are additive UI placement
decisions within an already-approved layout, not changes to the
architecture itself.
